Corona® Enterprise

Overview

Corona Enterprise allows you to extend the standard Corona APIs by adding your custom native extensions.

Lua already supports the creation of new libraries via the module/require mechanism. With Corona Enterprise, you can also add libraries that are implemented as native code. In other words, you can wrap native code behind a Lua interface so it looks just like any other Corona API (e.g. the 'display' library).

What's New

Starting in build 2012.979, the CoronaEnterprise folder has been reorganized. Specifically, the Enterprise projects have changed:

  • Enterprise projects can now sit outside the CoronaEnterprise folder tree.
  • Enterprise project templates are available that show how to share the same set of Corona assets (main.lua, images, etc) with both the iOS and Android Enterprise projects.

These changes enable easy drop-in replacements for new daily builds. You just copy and replace the old CoronaEnterprise folder.

NOTE: If you are using an older version, you must migrate your projects to use the latest version of CoronaEnterprise. See 'Migrating from older versions of CoronaEnterprise' below.

Prerequisites

You should have Mac OS X 10.7.x (Lion) installed on your computer.

In addition, for each target platform you will need the following installed:

  • Android: Android SDK revision 20.0.1 or higher, supporting Android platform 4.1 (API Level 16) or higher
  • iOS: XCode 4.5

Installation

Installation is fairly simple. You will need to complete 3 basic steps:

  • Copy the CoronaEnterprise folder to /Applications
  • Activate CoronaBuilder.app (see below)

Activation

You must purchase an Enterprise Native Extension subscription to activate this product. Contact sales via our web form for more information.

To activate, type the following:

/Applications/CoronaEnterprise/Corona/mac/bin/CoronaBuilder.app/Contents/MacOS/CoronaBuilder authorize email password

where email and password are associated with the Enterprise account subscription.

This will activate a computer to use CoronaBuilder. Every account is allowed one computer.

If you wish to activate CoronaBuilder on a different computer for the same account, you must first deactivate the original computer before activating on another one (or purchase additional subscriptions):

/Applications/CoronaEnterprise/Corona/mac/bin/CoronaBuilder.app/Contents/MacOS/CoronaBuilder deauthorize

Daily builds

If you wish to update to a new Enterprise daily build, you should simply overwrite the old CoronaEnterprise directory with the new one.

If you are using a version of Enterprise before 2012.979, you must migrate your older projects to the newer project format. See 'Migrating from older versions of CoronaEnterprise' below.

App Project Templates

The App project is a template project.

It creates a library called plugin.library that your Lua code can access. In the stub implementation, there is a library function show() that opens up a dictionary displaying the definition of the word passed to the function.

On iOS, it opens up a UIReferenceLibraryViewController. On Android, it opens up a web view to a dictionary web site.

New projects

When you create a new project, you should copy the App project template.

When you create a new plugin, you should also copy the App project template. In this case, the app serves as a test harness for your plugin.

Why all the changes?

The new App project template has several advantages over the older projects:

  • You may place the project outside the CoronaEnterprise folder tree.
  • The App project template allows you to develop for iOS and Android while simultaneously sharing the same set of Corona assets (e.g. main.lua, images, audio files, etc).
  • Enable you to create plugins that you can re-use across all your Enterprise projects.

Where files go

There are 3 top-level directories:

  • Corona This is where all Corona assets go (e.g. main.lua and other assets files like images, sounds, etc.).
  • android This is where the Android-specific code and Android project files go.
  • ios This is where the iOS-specific code and iOS project files go.

Android

The following are the main Android project files:

  • android/build.sh This is the main build script. The first argument should be the path to the Android SDK. If you have the ANDROID_SDK environment variable set, then no arguments are necessary.
  • android/AndroidManifest.xml This is where you set up application settings such as name, version, permissions, etc.
  • android/project.properties This is where you include the Corona library and optionally 3rd party libraries such as InMobi, inneractive, etc.
  • android/libs This is where 3rd party (Corona) plugins, e.g. JAR files, that your app relies on should go.
  • android/src This is where your Java source files go. Because the package is com.mycompany.app, this is where the files are located by default. You should modify this according to the package name you specify.

For plugins, there are additional files of interest:

  • android/build.plugin.sh This is the build script that creates the plugin. The output is a JAR file. The only files included in the JAR are those located in src/plugin. This script takes the same arguments as build.sh. You only need to invoke this script if you want to build the plugin separately.
  • android/AndroidManifest.plugin.xml This is where you set up the plugin's package name.
  • android/src/plugin/library/LuaLoader.java This is the code for the Lua library plugin.library on the Android side.

Walkthrough

We briefly outline the flow of the sample so you get a sense for how the pieces fit together:

  1. android/src/com/mycompany/app/CoronaApplication.java
    • At launch time, CoronaApplication is instantiated. It adds a private class to be notified of various app-level events like app suspend and resume.
    • Right before main.lua is invoked, the willLoadMain: method is invoked.
    • At this point, OpenGL is setup and all Corona frameworks are available.
  2. Corona/main.lua
    • In the Lua code, plugin.library is require'd. The Corona engine will then look for a corresponding Java class called plugin.library.LuaLoader and invoke it.
    • The name of this class is dynamic and is constructed using the original library name passed to require as the package name for a LuaLoader class. The LuaLoader class is expected to implement the JNLua interface com.naef.jnlua.JavaFunction.
  3. android/src/plugin/library/LuaLoader.java
    • When the fully qualified class path for LuaLoader is resolved, the class is instantiated with the default constructor. The invoke() method is then called by Lua via JNLua and does all the heavy lifting, e.g. creates the Lua library table, registers the Lua methods (e.g. show()), and then leaves the table at the top of the Lua stack.
    • The LuaLoader class is instantiated once for the lifetime of the process, much like a DLL or .SO file is typically loaded once for the lifetime of the process.
    • The invoke() method is called once for each Lua state that does a require. This corresponds to being called once each time the CoronaActivity is instantiated.

Building with Ant

Ant is a command line tool that is included with the JDK (Java Development Kit). It allows you to build your app via the command line. Follow the below steps to build App from the command line.

  1. On a Mac, open a Finder window.
  2. Go to the "Applications/Utilities" directory.
  3. Double click on the "Terminal" application.
  4. In the Terminal, change to the "./CoronaEnterprise/Samples/SimpleLuaExtension/android" directory via the cd command.
  5. Build the app by typing in ./build.sh <PathToAndroidSdkDirectory> where you would replace <PathToAndroidSdkDirectory> with the path to where you've downloaded the Android SDK to.

Here is an example of what you would type at the command line for the above:

cd "./CoronaEnterprise/Samples/SimpleLuaExtension/android"
./build.sh "~/android-sdk"

The "build.sh" script file will automatically update all Android project directories to use the Android SDK on your local machine via the android update project and android update lib-project command before building the project via the ant command line tool. This is a necessary step because ant cannot build the sample project until the project directory and all of its library project directories have been assigned a "local.properties" file by the android update command line tool.

The above will generate an unsigned APK in the SimpleLuaExtension/android/bin directory. You will need to sign this APK file before you can install it on a device. Please consult the Android documentation to learn more.

iOS

The following are the main iOS project files:

  • ios/App.xcodeproj This is the main project for building and running your app within Xcode.
  • ios/AppCoronaDelegate.h and AppCoronaDelegate.mm The main application entry points go here. The class conforms to the CoronaDelegate protocol so you will get notified immediately before and after main.lua is loaded and executed. You can also add UIApplicationDelegate methods. We now recommend that you add new Lua libraries via plugins (older projects would add them in this file).

For plugins, there are additional files of interest:

  • ios/Plugin.xcodeproj This is the main project that contains the Lua library that you are exposing to your application. This is a dependency of App.xcodeproj so it will get built automatically. You only need to open this project if you want to build the plugin separately.
  • ios/Plugin This is where the implementation of plugin.library goes.
  • ios/Plugin/PluginLibrary.h and Plugin/PluginLibrary.mm: This implements the plugin.library.

Walkthrough

We briefly outline the flow of the sample so you get a sense for how the pieces fit together:

  1. ios/main.mm
    • At launch time, MyCoronaDelegate is registered as the class implementing the CoronaDelegate interface.
    • Corona will instantiate an instance of this class. It assumes the designated initializer is init.
  2. ios/AppCoronaDelegate.mm
    • Right before main.lua is invoked, the willLoadMain: method is invoked.
    • At this point, OpenGL is setup and all Corona frameworks are available.
  3. Corona/main.lua
    • In the Lua code, plugin.library is require'd. The Corona engine will then look for a corresponding C function called luaopen_plugin_library and invoke it.
    • The name of this function is dynamic and is constructed using the original library name passed to require. It takes that name and adds a prefix luaopen_ to it, and then replaces . with _.
  4. ios/Plugin/PluginLibrary.mm
    • When luaopen_plugin_library is invoked, it called PluginLibrary::Open which does all the heavy lifting, e.g. creates the Lua library table, registers the Lua methods (e.g. show()), and then leaves the table at the top of the Lua stack.

Migrating from older versions of CoronaEnterprise

Starting with 2012.979, this project is designed to support the new organization of the CoronaEnterprise directory.

If you have projects based on prior versions of CoronaEnterprise, you'll need to update your projects:

  1. Create a new project by duplicating the App project.
  2. Copy the sources from your old project into this new project:
    • For iOS, copy your source files into the ios subdirectory, as appropriate.
    • For Android, copy your source files from the old project's src directory into the corresponding src of the new project.

NOTE: Per the above instructions, we highly recommend you create new projects from the App project template. Going the other direction (i.e. copying the new App project files into an older existing project) is not recommended.

Custom install directory for CoronaEnterprise

The default installation location is assumed to be: /Applications/CoronaEnterprise.

You can customize this location for your projects on a per-target platform basis:

  • For iOS, there is a symlink in the ios folder called CoronaEnterprise that points to /Applications/CoronaEnterprise by default. You can change this to the custom install location for CoronaEnterprise.
  • For Android, you must call the build.sh script with 2 parameters. The first is the path to the Android SDK (by default this is the ANDROID_SDK environment variable). The second is the path to CoronaEnterprise (by default this is /Applications/CoronaEnterprise).
